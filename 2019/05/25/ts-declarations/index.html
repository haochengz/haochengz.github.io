# 变量声明

let和const是Javascript中两个相对较新的变量声明方式. 之前提过的是, let和var某些方面较为接近, 但也避免了用户在Javascript中遇到的常见错误. const是let的增强, 其避免了重新给变量赋值.

既然Typescript是Javascript的超集, 也就自然的支持了let和const. 接着我们会仔细的观察这些新的声明以及为什么它们比var更好.

如果你对Javascript有些生疏了 那么下面一节将帮你回忆一下. 如果你非常熟悉Javascript中var声明的很多怪异行为, 那么你可以跳过接下来一节.

看到这里你可能已经意识到var的问题了, 这就是let语句被引入的原因. 除了使用了不同的关键字外, let语句的写法和var语句相同.

接着我们就会看到, 其关键的区别并非语法区别而是语义区别.

对于TypeScript来说，变量不一定都需要块作用域来告诉我们存在问题。

这也并不是说块作用域变量永远不能和函数作用域变量同时声明. 块作用域变量只是需要被声明在一个不同的块内.

在嵌套域中引入新名字的行为成为覆盖. 这是一把双刃剑, 在意外覆盖的情况下可能会引入某些的bug, 同时也避免了某些bug. 例如想象我们使用let改写之前的sumMatrix函数.

这个版本的循环实际正确的计算了矩阵的和, 因为内层循环的i覆盖了外层循环的i.

覆盖在编写清晰代码是应该避免. 而在某些场景下它可能非常适合, 此时应该运用你的判断力自行决定.

块作用域变量捕获

我们之前首次接触var声明的变量捕获时, 我们简短的介绍了捕获后的变量的行为. 为了使这个概率更符合直觉, 每次运行进入一个域时, 它就创建了一个变量环境. 这个环境和其所捕获的变量能一致存在就算所属的域已经执行结束.

由于我们在环境中捕获了city, 我们依然可以执行它就算它所位于的if块已经执行完毕.

回忆我们之前的setTimeout实例, 我们通过使用IIFE在for循环的每次迭代中捕获到变量状态. 实际上, 我们做的是为我们捕获的变量创建了一个新的变量环境. 这并不友好, 但幸运的是在Typescript中不需要再这么做了.

let声明在循环中编写非常不一样. 其并非每次迭代时引入新的环境, 而是每次迭代时创建新的作用域. 尽管这就是我们使用IIFE所做的, 我们可以使用let声明修改旧的setTimeout示例.

const声明

const声明是另一种声明变量的方法.

就像let声明, 但就像它的名字所暗示的, 它们所声明的变量绑定之后不能再被赋值. 换句话说, 它们和let具有相同的作用域规则, 但你不能重新赋值.

let vs. const

我们现在有两种具有相似作用域语义的声明方式, 也就很自然的会问应该用哪一种. 和其他宽泛的问题一样, 答案是: 看情况.

根据最小特权原则, 所有的声明, 除了那些你一开始就计划要做修改的外都应该使用const. 逻辑是那些你并不准备修改的变量, 其他工作在同一代码库的人不应该自动的获得修改权限, 同时需要认真的考虑需要修改这个变量吗? 使用const同时使代码在数据流中推导时更容易预测.

你应该自行判断, 如果情况适用于const, 和团队成员沟通.
