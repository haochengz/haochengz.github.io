纯函数的好处

再次强调”纯”

首先什么是纯函数?

纯函数是这样一种函数, 即相同的输入, 永远会得到相同的输出, 而且没有任何可观察的副作用.

比如slice和splice这两个函数, 它们作用类似但各自的方式却不大相同. slice函数符合纯函数的定义因为它确保了相同的输入能得到相同的输出. 而splice却会修改传入的那个数组; 这样就产生了可观察到的副作用, 即这个数组永久的改变了.

在函数式编程中我们需要的是可靠的, 每次都能返回相同结果的函数, 而不是会改变数据的函数.

在上面例子那个不纯的版本中, checkAge的结果取决于外部变量minimum. 也就是说它取决于系统状态; 由于引入了外部环境, 从而增加了认知负荷.

另一方面使用纯函数的形式, 函数就能做到自给自足. 也可以使minimum成为一个不可变对象, 这样就保留了纯粹性. 为了实现这个效果可以这样做:

副作用有哪些

现在来研究一下副作用. 首先看作用, 我们可以理解为一切除结果计算之外发生的事情. 那么副作用呢?

副作用是在计算结果的过程中, 系统状态的一种变化, 或者与外部世界进行的可观察的交互.

那么副作用就包括但不限于以下所列:

  • 更改文件系统
  • 往数据库插入记录
  • 发送一个http请求
  • 改变数据
  • 打印/Log
  • 获取用户输入
  • DOM查询
  • 访问系统状态

这个列表还可以继续写下去. 概括的说, 只要跟函数外部环境有交互的行为都是副作用. 这可能会让你怀疑无副作用编程的可能性. 但函数式编程的只要哲学就是假定副作用是造成不当行为的主要原因. 不过这并不是说要禁止一切副作用, 后面会说到functor和monad, 这将教会我们如何控制副作用. 我们现在来了解一下为什么我们坚持相同输入获得相同输出原则. 这就需要复习一下八年级数学了.

可移植性/自文档化

纯函数自给自足, 它所需要的东西都能轻易获得. 纯函数依赖明确, 便于观察和理解, 没有任何小动作.

我最喜欢的名言之一是 Erlang 语言的作者 Joe Armstrong 说的这句话：“面向对象语言的问题是，它们永远都要随身携带那些隐式的环境。你只需要一个香蕉，但却得到一个拿着香蕉的大猩猩…以及整个丛林”。

可测试性

纯函数使测试更加容易. 我们不需要伪造一个支付网关, 或者每次测试开始之前都要配置, 之后断言状态. 只需简单的给函数一个输入, 然后断言输出就好了.

合理性

很多人相信纯函数最大的好处是引用透明性. 如果一段代码可以替换成它所执行得到的结果, 而且是在不改变整个程序行为的前提下替换的, 那么我们说这段代码是引用透明的. 而对于纯函数, 它总能保证相同输入得到相同结果, 这就保证了引用透明性.

decrement, isSameTeam和punch都是纯函数, 所以引用是透明的. 我们可以使用一种等式推导的技术来分析代码. 所谓等式推导就是一对一的替换, 手动执行相关代码, 借助引用透明特性来剖析代码:

等式推导对于分析, 理解和重构代码非常重要.

并行代码

最后决定性的一点, 我们可以并行任意纯函数. 因为纯函数根本不访问任何共享内存, 也不会引入副作用而进入竞争态.

总结

纯函数本身是不同凡响的, 我们将尽力将所有函数写为纯函数. 但如果手头没有一些工具支持的话, 纯函数写起来会比较费力. 我们需要到处传递参数, 因此通过下一篇介绍的curry新工具辅助我们更容易的实现纯函数.

基础类型

简介

为了使程序能够进行最基础的操作, 我们需要一些最基础的数据单元: 数字, 字符串, 结构, 布尔值等. 在Typescript中我们除了支持和Javascript中相同的所有类型, 还包括了更方便的枚举类型.

布尔值

这是最基础的一个数据类型, 其值只能是true/false, 在Typescript和Javascript中被称为布尔值.

数字

像在Javascirpt中一样, Typescript中的所有数字都是浮点值. 其类型就是number. number类型除了支持十六进制和十进制的字面量外, Typescript引入了ECMAScript 2015标准中的二进制和八进制字面量表示.

字符串

对于使用Javascript编写web页面或类服务器端程序而言另一个最基础的部分就是操作字符数据. 和多数语言类似, 我们使用string类型表示字符串数据类型. 和Javascript一样, Typescript使用双引号(“)或单引号(‘)包裹字符串的值.

也可以使用模板字符串, 它能跨越多行且解析内部的程序表达式. 这种字符串由反引号(`)包裹, 内部的表达式写法为: ‘${ expr }’.

数组

和Javascript一样, Typescript允许操作一组值的集合. 数组类型可以有两种方式定义, 第一是使用元素类型后跟’[]’用于表示定义了一个包含某种类型元素的数组.

另一种定义方式是使用数组泛型, ‘Array‘.

元组类型用于表示一个类型和长度均固定的数组, 且其中元素类型不必相同. 例如你可能想要表示string和number组成的一对值.

Any

我们在写应用程序是可能会遇到需要定义一个不知道类型的变量. 这样的值可能来自动态内容, 如一个第三方库. 此时我们希望能关闭类型检查同时使该值通过编译时检查. 因此我们使用any类型来标注这类值.

Any类型在处理已有Javascript代码时是一个强大的工具, 其可以在编译时逐渐的引入或关闭类型检查. 你可能认为’Object’也能完成类似工作, 就像在其他语言中那样. 然而’Object’类型的变量仅仅允许你将任何值赋值给它. 你不能随意的调用它们的方法, 就算是存在的方法.

Any类型在你已知一部分类型时通用有用. 如你可能需要一个数组, 但数组的类型混合了多重类型.

Void

void差不多和any类型相反: 没有任意类型. 你应该经常在不返回任何值得函数返回类型上见到它.

将一个变量声明为void类型没什么用处, 因为你只将’undefined’和’null’赋值给它.

Null 和 Undefined

在Typescript中, undefined和null分别有它们自己的类型即undefined和null. 和void类似, 它们单独存在时并没有多大意义.

默认情况下null和undefined是所有类型的子类型. 这表示你可以把null和undefined赋值给如一个number.

然而, 但使用’–strictNullChecks’编译标志后, null和undefined只能赋值给void和它们各自的类型. 这有助于避免很多常见错误. 如你想传递一个可能是string, null或undefined的值, 你就可以使用联合类型: ‘string | null | undefined’. 同样的, 稍后再讨论联合类型.

提议: 我们鼓励只要可能就使用 ‘–strictNullChecks’, 但出于本指南的某些目的, 我们假设这个编译选项是关闭的.

Never

never类型表示这个类型的值用于不会产生. 例如never是一个函数表达式或箭头函数表达式的返回值, 其永远只会抛出异常或者用于不会返回. 变量同样可以被定义为never类型只要其永远不被赋值.

never类型是所有类型的子类型, 可以赋值为任何类型. 但没有任何是never的子类型, 或被赋值为never(除了never本身). 即是Any都不能被赋值为never.

类型断言是一种告诉编译器”相信我, 我知道我在干什么”的方式. 类型断言就像其他语言中的类型转换, 但并不做特定的检查和重新组织数据. 其并不影响运行时, 仅仅针对编译器. Typescript假设你, 程序员, 自行做了类型检查.

关于let

到现在你可能已经注意到, 我们使用let关键字而不是你可能更熟悉的Javascript的var关键字. let实际上是Javascript的新组成, 而Typescript实现了它. 我们稍后会讨论细节部分, 但通过使用let减少了Javascript中的很多问题, 因此只要可能你应该一直使用let而不是var.

发表于 2019-05-23 | 更新于 2019-05-24 | 分类于 函数式编程

概念

和面向对象编程等范式一样, 函数式编程也是一种编程范式. 函数式编程并没有比面向对象更好, 或者更差. 它们都使用特定的手段解决工作中的问题, 某些场景下函数式编程非常适合, 相应的某些场景下使用函数编程范式并不会是最好的选择. Javascript是一种多范式语言, 现代语言多数是这样的. 使用这些现在编程语言可以使你面对问题选择合适的编程范式.

如果你之前并不熟悉函数式编程, 仅仅使用面向对象的方式去编写程序. 而且你已经写过一些中等规模的应用程序. 那么你很有可能在可变状态(mutable state), 无限制副作用(unrestricted side effects)和无原则设计(unprincipled design)中遇到了麻烦.

写代码需要遵循一定的原则, 例如我们已经有了很多耳熟能详的原则: DRY(Don’t repeat yourself), 高内聚低耦合(loose coupling high cohesion), YAGNI(Ya ain’t gonna need it), 最小意外原则(Principle of least surprise), 单一责任原则(Single responsibility)等等. 这些通用的原则适应范围广泛, 编码时应当时刻考虑它们.
